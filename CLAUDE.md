# Stock-ARM (Stock Analysis & Research Manager)

> **Personal financial analysis tool**: A-share focused, LLM-powered, multi-source data, one-click deployment.

---

## !! DEVELOPMENT PARADIGM — FIRST-CLASS AUTHORITY !!

**The development paradigm defined in `docs/paradigm/` is the supreme authority of this repository.**
All agents MUST read and comply with paradigm documents BEFORE any other action.
When any specification, standard, or requirement conflicts with the paradigm, **the paradigm wins**.

### Paradigm Document Priority (read in this order)

1. **`docs/paradigm/agent-roles.md`** — Agent role definitions: Planner, Builder, QA. Each agent's scope, inputs, outputs, and boundaries.
2. **`docs/paradigm/sprint-workflow.md`** — The sprint cycle: how features flow from spec to code to merge. Includes the complete flow diagram, iteration limits, and escalation rules.
3. **`docs/paradigm/quality-gates.md`** — Evaluation criteria, scoring rubric, test layers, and pass/fail thresholds. Nothing merges without passing these gates.

### Paradigm Core Rules (non-negotiable)

- **One spec, one sprint**: Each sprint implements exactly one feature spec. No bundling.
- **Self-check before QA**: Builder must pass all self-check items before submitting to QA.
- **3-round limit**: A feature gets at most 3 QA iterations. Failure after 3 rounds escalates to user.
- **CLAUDE.md is alive**: Builder must update the Module Checklist below after each successful merge.
- **No deletion**: Conventions in this file can only be appended or marked deprecated with reason. Never deleted.
- **Skills-first evaluation**: Before implementing any feature, search for existing skills/tools that can be reused. Record findings in `skills/registry.yaml`.

### Paradigm Enforcement (automated, cannot be bypassed)

The paradigm is not just documentation — it is **enforced by automated tooling**:

1. **Git pre-commit hook** (`scripts/hooks/pre-commit`):
   - 阻止提交敏感文件（.env, 密钥文件）
   - 检测硬编码API Key
   - 修改src/代码时提醒补充测试
   - 校验CLAUDE.md完整性
   - 违反任何一项 → **提交被拒绝**

2. **Git commit-msg hook** (`scripts/hooks/commit-msg`):
   - 校验提交消息格式: `<type>(<module>): 描述`
   - 格式不对 → **提交被拒绝**

3. **Claude Code hooks** (`.claude/settings.local.json`):
   - agent执行`git commit`前自动触发范式合规检查
   - agent写入`specs/`目录的文件时自动校验Feature Spec格式
   - 会话结束时自动校验CLAUDE.md状态

4. **验证脚本** (`scripts/validate_paradigm.py`):
   - `--all`: 全量验证（CLAUDE.md + 依赖 + 所有Spec格式）
   - `--spec <path>`: 校验Feature Spec的7个必需章节
   - `--sprint <path>`: 校验Sprint Report的7个必需字段
   - `--pre-commit`: 提交前综合检查
   - `--deps`: 依赖一致性检查
   - `--claude-md`: CLAUDE.md完整性检查
   - QA Agent评估前必须运行 `--all` 确认合规

## Coding Conventions

- **Python**: Black formatter, isort imports, type hints required on all public functions
- **TypeScript**: ESLint + Prettier
- **File naming**: Python `snake_case.py`, TypeScript `PascalCase.tsx` (components), `camelCase.ts` (utils)
- **No raw dicts**: Module boundaries use Pydantic Models
- **No hardcoded config**: All external values go through the config system
- **Chinese-first content**: 代码注释、文档、日志消息、错误提示、UI文本、Prompt模板等所有人类可读文本均使用中文（变量名、函数名等标识符除外）

## API Design Conventions

- RESTful routes: `/{resource}` (plural), nested: `/{resource}/{id}/{sub-resource}`
- Unified response: `{"code": 0, "data": {...}, "message": "ok"}`
- Error response: `{"code": <error_code>, "data": null, "message": "<human-readable>"}`
- Error codes: 1xxx=client error, 2xxx=data source error, 3xxx=LLM error, 4xxx=system error

## Database Conventions

- ORM: SQLAlchemy 2.0+ (async)
- Migrations: Alembic
- Table naming: `snake_case`, singular (e.g., `analysis_record`, `scheduler_state`)
- All tables must have `id` (primary key), `created_at`, `updated_at`
- **Storage principle**: Only persist data that cannot be re-fetched (user config, analysis results, scheduler state, LLM logs). Raw market data, news originals, and document files are NOT stored locally.
- Scheduled data cleanup: old records beyond retention period auto-deleted monthly

## Test Conventions

- Test files: `tests/unit/` and `tests/integration/`, mirroring `src/` structure
- Naming: `test_<module>.py`, functions `test_<behavior>_<scenario>()`
- External deps mocked in unit tests, real SQLite (in-memory) for integration
- Commands: `pytest tests/` (all), `pytest tests/unit/` (unit only)

## Configuration

- Priority: env vars > `.env` file > `config.yaml` defaults > code defaults
- Env var naming: `STOCK_ARM_<CATEGORY>_<NAME>` (e.g., `STOCK_ARM_LLM_API_KEY`)
- Secrets: only via env vars or `.env` (which is gitignored)
