# 设计需求

## 1. 产品定位

一个**深度定制的个人金融分析工具**，核心特征：
- **以A股分析为主**，国际市场（美股、港股）为辅助跟踪
- **以LLM API为分析核心**，结合量化技术和多源数据，自动完成信息采集、整理、分析、输出全流程
- **用户自用**，不面向公众，无合规负担，但需保障数据安全和API密钥安全
- **安装部署傻瓜化**，一键启动，零运维

## 2. 技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| 后端 | Python 3.11+ / FastAPI | 金融生态最成熟（akshare、tushare、efinance等），async支持好，LLM SDK丰富 |
| 前端 | React 18 + TypeScript + Vite | 组件生态丰富，图表库（ECharts/Recharts）支持好，reference项目已验证 |
| 数据库 | SQLite (WAL模式) | 零配置，单文件备份，个人工具无需分布式，跨平台兼容 |
| 缓存 | 内存缓存 (Python `cachetools`) | 轻量，无额外依赖，热数据（当日行情、实时报价）缓存于内存即可 |
| 移动端 | PWA + 消息Bot（飞书/微信） | PWA：浏览器安装即用，无需应用商店，响应式适配手机端；Bot：推送通知、快速查询 |
| 桌面端 | 浏览器直接访问（本地HTTP服务） | 最简方案，`python main.py` 启动后自动打开浏览器 |
| 容器化 | Docker + docker-compose | 统一开发/部署环境，消除依赖差异；用户一键 `docker-compose up` 即可运行 |
| 交易接口 | 仅设计抽象框架，不实现具体功能 | 预留 `TradeProvider` 抽象层，后续可接入Longbridge等券商API |

## 3. 设计内容

### 3.1 数据源与数据获取

#### 3.1.1 A股数据源（多源容灾，优先级降序）

| 优先级 | 数据源 | 能力 | 是否需要Token |
|--------|-------|------|--------------|
| 0 | efinance | 实时行情、K线、板块 | 否 |
| 0 | Tushare Pro | 行情、财务、宏观、分红 | 是（免费额度有限） |
| 1 | AkShare | 行情、宏观经济指标、板块、资金流向 | 否 |
| 2 | pytdx (通达信) | 实时行情、K线 | 否 |
| 3 | Baostock | 历史K线、基本面 | 否 |

**容灾策略**：按优先级依次尝试，某源失败自动降级到下一源。每次失败需记录具体错误信息（哪个源、什么接口、什么错误），方便排查。

#### 3.1.2 国际市场数据源

| 数据源 | 覆盖市场 | 能力 | 备注 |
|--------|---------|------|------|
| Longbridge OpenAPI | 美股、港股、新加坡 | 行情、K线、财务、新闻、持仓 | 需OAuth认证，参考 `/rvhome/hao.gao/personal/developers/skills/longbridge` |
| yfinance | 美股（辅助） | 历史K线、基本面 | 免费，无需Token |

#### 3.1.3 新闻与舆情数据源

| 数据源 | 用途 | 备注 |
|--------|------|------|
| Tavily Search API | 中英文新闻搜索 | 免费1000次/月 |
| SerpAPI | 百度/Google搜索 | 付费 |
| 雪球爬虫 | 个股讨论、板块消息 | 需反爬策略（随机UA、请求间隔、代理池） |
| 东方财富爬虫 | 研报、公告、资金流向 | 需反爬策略 |
| RSS订阅 | 财经媒体（华尔街见闻、财联社等） | 低成本持续跟踪 |

**反爬设计原则**：
- 请求间隔随机化（基础间隔 + 随机抖动）
- User-Agent 池轮换
- 支持配置代理池（可选）
- 请求频率自适应：触发反爬后自动降速或切换源
- 爬取失败不阻塞主流程，降级为"数据缺失"标注

#### 3.1.4 宏观经济数据

**需跟踪的核心指标**：

| 类别 | 指标 | 更新频率 | 数据源 |
|------|------|---------|--------|
| 物价 | CPI, PPI | 月度 | AkShare / Tushare |
| 货币 | M1, M2, 社融规模 | 月度 | AkShare |
| 利率 | LPR, Shibor, 国债收益率(10Y/1Y) | 每日/月度 | AkShare |
| 市场 | 股债利差(=1/全A PE - 10Y国债) | 每日 | 计算值 |
| 市场 | 两融余额 | 每日 | AkShare |
| 市场 | 北向资金净流入 | 每日 | AkShare / efinance |
| 市场 | 沪深港通额度使用 | 每日 | AkShare |
| 国际 | 美联储利率决议、非农、CPI(美) | 事件驱动 | 财经日历API |
| 商品 | 原油(WTI/Brent)、黄金、铜 | 每日 | AkShare / yfinance |

**宏观数据处理流程**：
1. 每日定时拉取 → 存入数据库 → 与前值/预期值对比
2. 生成趋势图表（折线图，标注关键转折点）
3. LLM分析当期数据含义及对市场影响
4. 若当日无更新，标注"暂无更新"并显示最近一次数据及日期
5. 推送摘要至Bot

### 3.2 LLM集成

#### 3.2.1 多模型支持

通过 **LiteLLM** 统一接入不同厂商API：

| 厂商 | 模型示例 | 用途 |
|------|---------|------|
| OpenAI | gpt-4o, gpt-4o-mini | 综合分析 |
| Anthropic | claude-sonnet-4-20250514 | 深度推理 |
| DeepSeek | deepseek-chat, deepseek-reasoner | 性价比 |
| 通义千问 | qwen-plus, qwen-max | 中文优化 |
| MiniMax | abab6.5s, abab7 | 中文长文本处理 |
| 智谱AI | glm-4-plus, glm-4-flash | 中文综合分析 |

**模型使用策略**：
- **主模型**：用于核心分析任务（个股分析、宏观解读）
- **快速模型**：用于批量摘要、新闻分类等轻量任务
- **多模型投票**（可选）：关键决策可配置多模型交叉验证
- **Fallback链**：主模型失败时自动切换备选模型

#### 3.2.2 LLM调用规范

- 所有Prompt使用模板管理，存放在 `prompts/` 目录，支持版本迭代
- 每次调用记录：模型名、token用量、耗时、成功/失败
- 输出结构化校验：LLM返回结果必须通过Schema验证，不合格则重试或降级
- 成本控制：可配置单日/单月token上限，超限告警

### 3.3 核心功能模块

#### 3.3.1 自选股管理与分析

**自选分组**：
- 支持创建多个自选组（如"长期持仓""短线关注""行业跟踪"）
- 每组可设置不同的分析策略和推送频率
- 支持从文本、CSV、图片（OCR识别股票代码）导入

**每日分析内容**：
- 技术面：均线趋势、MACD、RSI、量价关系、支撑/阻力位
- 基本面：PE/PB/ROE、最新财务数据摘要
- 消息面：相关新闻、公告、研报摘要
- 资金面：主力资金流向、筹码分布（可选）
- LLM综合研判：多空观点、关键风险提示、操作建议
- 与前一交易日对比：信号变化标注

#### 3.3.2 财报季跟踪

**触发条件**：每年4月（年报/一季报）、8月（半年报）、10月（三季报）为高频期

**处理流程**：
1. 每日扫描当日新发布的财报（AkShare/东方财富）
2. 自动下载PDF原文（可选）
3. LLM生成财报摘要：
   - 主营业务构成及变化
   - 核心财务指标：营收、净利润、扣非净利润、毛利率、净利率、经营现金流
   - 同比/环比变化及趋势
   - 亮点与风险提示
4. 以文件形式（Markdown）存储摘要，按日期/股票代码组织
5. 用户可标记"感兴趣"的财报，触发深度分析：
   - 分业务/分区域拆解
   - 横向（同行业对比）和纵向（历史趋势）分析
   - 估值合理性评估

#### 3.3.3 国际金融与地缘跟踪

**跟踪范围**：
- 美股三大指数（道琼斯、标普500、纳斯达克）每日走势及要闻
- 港股恒生指数、恒生科技指数
- 美联储、欧央行政策动态
- 地缘冲突（如中美关系、地区冲突）对市场的潜在影响
- 大宗商品异动（原油、黄金暴涨暴跌）
- 全球重大经济事件日历

**输出格式**：每日一份"国际金融简报"，包含：
- 隔夜美股/港股表现总结
- 关键事件及潜在A股影响分析
- 需关注的近期事件预告

#### 3.3.4 研报管理

**数据源**：
- 东方财富研报中心
- 券商APP研报入口（如有API）
- Longbridge研报（港美股）

**功能**：
- 按行业/个股/评级筛选和下载
- LLM摘要：核心观点、目标价、逻辑链、风险提示
- 研报观点跟踪：对比不同券商对同一股票的评级变化
- 推送匹配自选的研报

#### 3.3.5 平台信息采集

**目标平台**：
- 雪球：个股讨论、组合跟踪、大V观点
- 东方财富股吧：市场情绪感知
- 同花顺（如有可用接口）

**反爬策略（核心）**：
- 请求间隔：基础2-5秒 + 随机0-3秒抖动
- User-Agent池：维护50+真实UA，随机轮换
- 登录态管理：Cookie池（如需登录内容）
- 频率自适应：被拦截后指数退避（2s→4s→8s...），超过阈值暂停该源
- 数据缓存：同一内容24小时内不重复抓取
- 优雅降级：爬取失败时使用搜索引擎API作为备选

### 3.4 外部Skills与工具生态集成

#### 3.4.1 设计理念

各平台和社区（如Claude Skills、GitHub开源项目、量化社区等）存在大量成熟的金融分析、建模、数据处理相关skills和工具。本项目应当：

- **主动发现与评估**：在实现每个功能模块前，agent应主动搜索和评估是否有可直接使用或参考的现有skills/工具
- **标准化接入**：设计统一的 `SkillAdapter` 接口层，使外部skills能以统一方式被调用
- **轻量依赖原则**：集成外部skills时，**不得引入重量级框架或大量额外依赖**。优先选择纯Python实现、依赖少的工具。如某skill需要引入超过3个新pip包，需评估是否值得，或考虑仅参考其设计思路自行实现
- **分级集成策略**：
  - **直接使用**：成熟、依赖少、匹配度高的skills，直接接入（如Longbridge的MCP接口）
  - **参考借鉴**：设计优秀但依赖过重或不能直接使用的，提取其核心逻辑和Prompt策略，轻量化实现
  - **二次封装**：功能匹配但接口不兼容的，编写薄适配层后集成
- **持续迭代**：建立skills注册表（`skills/registry.yaml`），记录已集成、已评估、待评估的skills，随项目迭代持续更新

#### 3.4.2 skills集成的具体方向

| 方向 | 可集成的skills类型 | 集成方式 |
|------|------------------|---------|
| 技术分析 | K线形态识别、缠论分析、波浪理论 | 参考/二次封装 |
| 量化策略 | 动量策略、均值回归、多因子模型 | 参考设计，适配YAML策略格式 |
| 数据获取 | 券商API skills、金融数据爬虫 | 直接使用/适配 |
| LLM分析 | 金融Prompt模板、研报解读skills | 参考Prompt设计 |
| 可视化 | 图表生成、仪表盘模板 | 参考/直接使用 |
| 风控 | 风险评估模型、仓位管理计算 | 二次封装 |

#### 3.4.3 agent在实现时的skills评估流程

```
实现新功能前:
  1. 搜索相关skills/工具（GitHub、Claude Skills目录、量化社区）
  2. 评估匹配度、质量、维护状态、**依赖体量**
  3. 决策：直接用 / 参考 / 忽略
     - 直接用的前提：依赖少（≤3个新pip包）、接口清晰、维护活跃
     - 依赖过重或维护停滞的：仅参考核心逻辑，自行轻量实现
  4. 记录评估结论到 skills/registry.yaml
  5. 如采用，编写薄适配层并集成测试（适配层代码不超过200行）
```

### 3.6 网络与错误处理

**网络访问层设计**：

```
所有外部HTTP请求统一经过 NetworkClient：
  - 超时配置：连接超时5s，读取超时30s（可按接口定制）

      "source": "AkShare",

      "error_type": "ConnectionTimeout",
      "error_msg": "连接超时(5s)",
      "suggestion": "请检查网络连接或配置代理"
    }
  - 错误聚合：多次同类错误合并报告，避免日志洪泛
```

#### 3.7.1 Docker环境设计

Docker在本项目中的定位是**开发环境统一化**和**可选的部署方式**，核心思路：
- 提供一个标准Docker配置，开发者可直接用于开发，确保环境一致
- 支持检测用户本地已有的Docker环境，自动补全缺失的配置和依赖

**容器架构**：
```
docker-compose.yml（开发与部署统一）
  ├── app          # 全栈服务（Python后端 + 前端静态文件）
  └── volumes:
        ├── ./data:/app/data         # SQLite数据库持久化
        ├── ./config:/app/config     # 用户配置持久化
        ├── ./logs:/app/logs         # 日志持久化
        └── ./src:/app/src           # 开发模式：源码挂载，热重载
```

**Docker配置要求**：
- 提供单一 `Dockerfile`，multi-stage build（构建前端 → 安装Python依赖 → 运行阶段）
- 基础镜像使用精确版本tag（如 `python:3.11.9-slim`），不使用 `latest`
- 前端在构建阶段编译为静态文件，由后端直接托管（FastAPI StaticFiles），无需独立Nginx容器
- 数据卷挂载确保数据库、配置、日志在容器重建后不丢失
- `.env` 文件通过 `env_file` 注入，不烘焙到镜像中
- 提供 `docker-compose.override.yml` 用于开发模式覆盖（挂载源码 + `uvicorn --reload`）

#### 3.7.2 环境检测与自适应

启动时自动检测运行环境并适配行为：

```
环境检测流程：

  1. 检测运行模式
     ├── 检查是否在Docker容器内（/.dockerenv 或 cgroup）
     │     ├── 是 → Docker模式
     │     └── 否 → 本地模式
     └── 各模式行为差异：
           Docker模式：数据目录 /app/data，日志同时写文件和stdout，
                       不启动浏览器，监听 0.0.0.0
           本地模式：数据目录 ./data，日志写文件，
                     启动浏览器，监听 127.0.0.1

  2. 检测本地Docker环境（仅本地模式下执行）
     ├── 检测Docker是否已安装（docker --version）
     ├── 检测Docker Compose是否可用
     ├── 检测是否已有本项目的Docker镜像/容器
     │     ├── 已有 → 提示用户可直接 docker-compose up 启动
     │     └── 未有 → 继续本地模式启动
     ├── 检测已有Docker环境的配置完整性
     │     ├── 缺少 .env → 从 .env.example 生成并提示用户填写
     │     ├── 缺少 data 卷目录 → 自动创建
     │     └── 镜像版本过旧 → 提示用户 docker-compose pull 更新
     └── 输出Docker环境状态摘要

  3. 检测Python版本和关键依赖（本地模式）
     ├── 版本不满足 → 明确报错并提示最低版本要求
     └── 依赖缺失 → 提示安装命令或建议使用Docker

  4. 检测网络环境（所有模式）
     ├── 测试关键数据源可达性（AkShare、Tushare等）
     ├── 测试LLM API可达性
     └── 不可达的源 → WARNING日志 + 标记为不可用，不阻塞启动
```

**环境变量兜底**：
- `RUNTIME_ENV=docker|local|auto`（默认auto，自动检测）
- `DATA_DIR`（覆盖默认数据目录）
- `LOG_TO_STDOUT=true|false`（Docker下默认true）
- `BIND_HOST=0.0.0.0|127.0.0.1`（Docker下默认0.0.0.0）

# 项目构建的开发范式

## 1. 核心理念

本项目采用 **多Agent协作开发范式**，目标是使AI Agent能够**持久、准确、可预测地**进行自动化开发。范式的关键不是让agent"聪明"，而是让流程**机械化、可重复、可审计**。

## 2. Agent角色定义

### 2.1 规划器（Planner Agent）

**职责**：产品设计与技术架构决策
**输入**：用户需求 / 功能描述 / Bug报告
**输出**：功能规格文档（Feature Spec）

**工作规范**：
- 只关注"做什么"和"为什么"，不关注"怎么做"
- 每个功能规格必须包含：
  ```
  ## 功能名称
  ## 用户故事（谁/想要什么/为了什么）
  ## 验收标准（可测试的条件列表）
  ## 数据流（输入→处理→输出）
  ## 接口契约（API路由、请求/响应Schema）
  ## 依赖项（依赖哪些已有模块）
  ## 非功能需求（性能、安全、兼容性）
  ## Skills评估（是否有可集成的现有skills/工具，评估结论）
  ```
- 规格粒度：一个规格 = 一个可独立交付的功能增量
- 规格存放路径：`specs/` 目录，按模块分类

### 2.2 生成器（Builder Agent）

**职责**：功能实现
**输入**：功能规格文档
**输出**：代码 + 单元测试 + 更新的CLAUDE.md

**工作流程——冲刺循环（Sprint Cycle）**：

```
每个冲刺处理一个功能规格：

1. 准备阶段
   ├── 阅读功能规格，确认理解
   ├── 阅读CLAUDE.md，了解现有架构和约定
   ├── 搜索和评估可复用的现有skills/工具
   └── 确认依赖模块已就绪

2. 实现阶段
   ├── 编写代码（遵循CLAUDE.md中的编码规范）
   ├── 编写单元测试（覆盖验收标准中的每个条件）
   ├── 运行lint和类型检查
   └── 运行单元测试，确保全部通过

3. 自检阶段（提交评估前）
   ├── 代码是否符合CLAUDE.md编码规范？
   ├── 是否引入了新的依赖？如是，是否已更新requirements.txt？
   ├── 是否所有验收标准都有对应测试？
   ├── 是否有硬编码的配置值？（应使用配置系统）
   ├── 错误处理是否完整？外部调用是否有超时和重试？
   ├── 是否有安全问题？（API Key暴露、SQL注入等）
   └── 若集成了外部skills，适配层是否有独立测试？

4. 提交评估
   └── 输出冲刺报告，交给评估器
```

**冲刺报告格式**：
```
## 冲刺报告
- 功能规格：[规格名称]
- 新增/修改文件列表
- 新增依赖列表
- 测试覆盖情况（通过/总数）
- 自检清单结果（逐项通过/不通过）
- 已知限制或遗留问题
- 集成的外部skills（如有）
```

### 2.3 评估器（QA Agent）

**职责**：质量把关与测试验证
**输入**：冲刺报告 + 功能规格 + 代码变更
**输出**：评估报告（通过/不通过 + 详细反馈）

**评估维度与评分标准**：

| 维度 | 权重 | 通过标准 |
|------|------|---------|
| 功能完整性 | 30% | 所有验收标准对应的测试均通过 |
| 代码质量 | 20% | lint无error，类型检查通过，无明显code smell |
| API契约 | 15% | 接口路由、请求/响应格式与规格一致 |
| 错误处理 | 15% | 外部调用有超时/重试/降级，错误信息清晰 |
| 数据库 | 10% | Schema变更有migration，数据完整性约束正确 |
| 安全性 | 10% | 无硬编码密钥，无注入风险，输入有校验 |

**评分规则**：
- 总分 ≥ 80分：通过，可合并
- 总分 60-79分：条件通过，需修复指定问题后复评
- 总分 < 60分：不通过，需重新实现

**评估器的测试层级**：
```
1. 单元测试（生成器已写）：复核覆盖率和用例质量
2. 集成测试（评估器补充）：
   ├── API端到端测试（HTTP请求→响应校验）
   ├── 数据源容灾测试（模拟主源失败，验证降级）
   └── 数据库状态测试（操作前后数据一致性）
3. 场景测试（评估器补充）：
   ├── 正常场景（Happy Path）
   ├── 边界场景（空数据、超大数据、特殊字符）
   └── 异常场景（网络断开、API限流、格式错误）
4. 环境测试（评估器补充）：
   ├── Docker构建测试（镜像能否正常构建）
   ├── Docker运行测试（容器启动后功能是否正常）
   └── 环境检测测试（Docker/本地模式自动识别是否正确）
```

## 3. 流程治理

### 3.1 CLAUDE.md 管理规范

CLAUDE.md 是项目的**活文档**，是agent理解项目的核心依据。

- Python 3.11+, FastAPI 0.110+, React 18, TypeScript 5
- 关键依赖及版本约束

# 测试约定

- Mock规则

# 配置管理

- 配置优先级

**更新规则**：
- 每次冲刺通过评估后，生成器必须更新CLAUDE.md的模块开发清单

# 设计准则

## 1. 环境与依赖管理

- 使用 `pyproject.toml` 管理Python项目元数据和依赖
- `requirements.txt` 锁定精确版本（`pip freeze`生成）
- 前端使用 `package-lock.json` 锁定版本
- Python版本要求通过 `.python-version` 文件声明
- 每次新增依赖必须同步更新锁文件并记录在冲刺报告中
- Docker镜像版本锁定：`Dockerfile` 中基础镜像使用精确tag（如 `python:3.11.9-slim`），不使用 `latest`
- 新增Python依赖后，必须验证Docker镜像能正常构建（避免需要系统级编译依赖的包在slim镜像中构建失败）
- 可执行文件打包：重大版本更新后需验证Windows和macOS的打包产物能正常运行

## 2. 配置管理

- **零硬编码原则**：所有外部地址、密钥、阈值、间隔时间均通过配置系统管理
- 配置优先级：环境变量 > `.env` 文件 > `config.yaml` 默认值 > 代码内默认值
- 敏感信息（API Key等）只能通过环境变量或`.env`文件配置，`.env`文件加入`.gitignore`
- 提供 `.env.example` 模板，注释说明每个配置项的用途和格式

## 3. 错误处理与可观测性

- 所有外部调用（HTTP请求、数据库操作、LLM调用）必须有错误处理
- 错误信息必须包含：发生位置、操作内容、具体错误、用户可操作的建议
- 日志分级：DEBUG（开发调试）、INFO（正常流程）、WARNING（降级/重试）、ERROR（功能异常）
- 关键操作（数据源切换、LLM调用结果、推送状态）必须有INFO级日志

## 4. 数据安全

- API密钥不得出现在代码、日志、前端页面中
- 数据库文件存放在 `data/` 目录，加入 `.gitignore`
- 前端通过后端代理访问所有外部API，不直接暴露密钥到浏览器

## 5. 模块化与解耦

- 每个模块通过接口/抽象类定义契约，不依赖具体实现
- 数据源：`BaseDataFetcher` → 具体Fetcher
- LLM：`BaseLLMProvider` → LiteLLM统一层
- 通知：`BaseNotifier` → 具体渠道实现
- 交易：`BaseTradeProvider` → 具体券商实现（预留）
- Skills：`BaseSkillAdapter` → 具体skill适配器
- 模块间通过明确的数据结构（Pydantic Model）传递数据，不传递原始dict

## 6. 测试规范

- 测试文件位于 `tests/` 目录，镜像源码结构
- 每个公共函数至少一个正向测试、一个异常测试
- 外部依赖（API调用、数据库）在单元测试中使用mock
- 集成测试使用独立的测试数据库（SQLite内存模式）
- 测试命令标准化：`pytest tests/` 运行全部，`pytest tests/unit/` 运行单元测试

## 7. 前端设计准则

- 响应式设计，同时适配桌面端和移动端（PWA）
- 图表组件统一使用ECharts（金融图表生态最成熟）
- 页面状态管理使用Zustand（轻量，符合个人工具定位）
- API调用统一封装在 `src/api/` 目录
- 深色/浅色主题支持（金融工具用户偏好）

## 8. Skill沉淀规范

- 开发过程中产生的可复用能力（分析策略、数据处理流程、Prompt模板，测试构建，建模策略）应沉淀为独立skill
- Skill定义使用YAML格式，存放在 `skills/` 目录
- 每个skill包含：名称、描述、触发条件、输入/输出Schema、实现逻辑引用
- 外部集成的skill统一登记在 `skills/registry.yaml`，标注来源、版本、适配状态
- 自动生成的skill应包含足够的注释，说明其用途和使用场景

## 9. 性能与资源约束

- 单次全量分析（自选股+宏观+简报）应在合理时间内完成
- 数据源请求并发控制：默认3个并发worker，可配置
- LLM调用间隔：避免触发速率限制，可配置延迟
- 内存缓存设置TTL和最大条目数，防止内存泄漏
- SQLite使用WAL模式，支持读写并发
